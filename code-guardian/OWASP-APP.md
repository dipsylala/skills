# OWASP Top 10:2025 - Traditional Web/Application Detection Guide

For each risk: signals to detect in generated code, and the fix principle to apply.

Source: <https://owasp.org/Top10/2025/>

| # | Risk | Detect | Fix principle |
| --- | ------ | -------- | --------------- |
| A01 | Broken Access Control | Privileged action with no auth check; user-supplied path, URL, or ID used without validation; missing ownership or tenant check; CSRF on state-changing routes; server-side fetch from attacker-controlled URL | Validate identity and authorization before action; verify ownership/tenant boundaries; canonicalize and bound-check paths; require CSRF protection; allowlist outbound fetch targets |
| A02 | Security Misconfiguration | Debug mode, stack traces, or verbose errors in responses; CORS `*` on sensitive routes; default credentials; permissive headers; disabled security middleware | Default to production-safe config; restrict CORS to known origins; reject placeholder secrets at startup; enable framework security headers and middleware |
| A03 | Software Supply Chain Failures | Dependency pinned to `*`, `latest`, branch, or untrusted source; lockfile disabled; install/build script added without need; unauthenticated artifact download; deprecated or unmaintained component | Pin versions and keep lockfiles; use trusted registries; verify checksums/signatures when available; minimize install scripts; run SCA/dependency scanning |
| A04 | Cryptographic Failures | Secret or credential as string literal; MD5/SHA1/DES/RC4/ECB usage; plaintext sensitive storage; sensitive data in logs; password hashed with a fast general-purpose hash | Load secrets from environment or secret store; use authenticated encryption such as AES-GCM or ChaCha20-Poly1305; use Argon2id, bcrypt, scrypt, or PBKDF2 for passwords; use HMAC for keyed integrity; never log sensitive values |
| A05 | Injection | User input interpolated into SQL, shell, HTML, XML, LDAP, NoSQL, or template strings; dynamic use of `eval`, `exec`, `system`, or shell invocation; unsafe template rendering | Use parameterized queries/prepared statements; avoid shells and dynamic execution; pass arguments as structured arrays; escape or encode output for the target context |
| A06 | Insecure Design | No validation at trust boundary; insecure default (`debug=true`, `allow_all`); optional field omission bypasses logic; missing rate limit or replay protection for sensitive flows | Validate and parse at every trust boundary; default deny; make required state explicit; add rate limits, replay checks, and abuse controls for sensitive workflows |
| A07 | Authentication Failures | Password stored or compared as plaintext; weak password reset flow; token generated with weak PRNG (`Math.random`); JWT/session without expiry; auth bypass on missing/empty credentials | Use proven auth framework patterns; hash passwords with Argon2id/bcrypt/scrypt/PBKDF2; use cryptographic random token generation; set expiry, rotation, and invalidation; reject missing credentials explicitly |
| A08 | Software or Data Integrity Failures | Untrusted data deserialized without schema validation; user input spread/assigned directly onto a model; unsigned update/plugin/artifact accepted; webhook accepted without signature check | Validate schemas before deserializing; use explicit allowlist assignment; verify signatures/checksums for artifacts, updates, and webhooks; keep trust boundaries explicit |
| A09 | Security Logging and Alerting Failures | Auth events, privilege changes, high-risk access, or destructive operations not logged; errors silently swallowed; logs omit actor/resource/outcome; security events logged without alert path | Log security-relevant events with actor, resource, action, and outcome; sanitize sensitive data; surface errors safely; connect high-risk events to alerting or monitoring |
| A10 | Mishandling of Exceptional Conditions | Exception path fails open; missing parameter or null state bypasses checks; partial update leaves inconsistent state; caught error ignored; fallback grants access or uses unsafe defaults | Fail closed; validate required parameters and state; use centralized error handling; roll back partial work; return sanitized errors; preserve enough detail in logs for diagnosis |
